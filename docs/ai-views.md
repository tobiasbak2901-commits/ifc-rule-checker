# AI Views

## Purpose

`AI Views` in Object Tree is a task-focused workflow view, not a raw IFC hierarchy.

It presents:

1. Model health summary
2. Prioritized cards
3. Actionable details with selection hooks into the 3D viewer

## Card Order

Cards are rendered in this priority order:

1. `Clashing elements`
2. `Unclassified elements`
3. `High-risk systems`
4. `Recently selected` (when available)

Each card includes:

- icon + title
- count badge
- short explanation (`why it matters`)
- primary CTA
- secondary CTA
- expandable details list

## Model Health Scoring

Health summary is computed from derived counts:

- start at `100`
- subtract clash penalty (scaled, capped at `40`)
- subtract unclassified penalty (scaled, capped at `30`)
- subtract `10` if any high-risk buckets exist
- clamp to `0..100`

Status labels:

- `Good` (>= 80)
- `Needs attention` (50..79)
- `Critical` (< 50)

Bullets are clickable and target/expand relevant cards.

## DisplayName Rules

Element labels in AI Views are generated by `get_display_name(element)`.

Rules:

1. Use a meaningful element name if available and safe.
2. Reject GUID-like/default/technical names (`Default`, `Port_*`, GUID formats, raw IFC-like strings).
3. For known MEP types, generate concise human labels:
   - `IfcPipeSegment`/`IfcFlowSegment` -> `Pipe Ø<mm>` (if diameter found)
   - `IfcDuctSegment` -> `Duct <W>x<H>` (if available)
   - `IfcCableCarrierSegment` -> `Cable tray <W>x<H>`
   - `IfcFlowFitting` -> `Fitting`
   - `IfcValve` -> `Valve`
   - `IfcPump`/`IfcFan` -> `Equipment`
4. Fallback to friendly IFC label (never GUID).

## Data Sources

Cards and counts are built from:

- IFC element index (`state.ifc_index`)
- clash issues (`state.bcf_issues`)
- classification labels + overrides (`elem.class_name`, `_class_overrides`)
- recent selection history

Implementation entry points:

- model builder: `ui/panels/ai_views_model.py`
- cards UI: `ui/panels/ai_views_panel.py`
- host integration: `ui/panels/object_tree_panel.py`
